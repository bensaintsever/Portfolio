01 HUSTLE UP

Designing with AI

Grafting a mockup generation workflow onto the design system, without relinquishing control.

TL;DR

A workflow where AI generates mockups anchored in the existing design system (tokens + real components), not "in a vacuum". Result: accelerated production without loss of consistency, the designer retains decision-making power at every step.

Setting up the AI-assisted design workflow
Figma, Figma MCP, Claude Code, Claude Design (POC)
Based on the multi-audience, multi-theme DS
late 2024
AI WORKFLOW ANCHORED IN DS

The workflow: an established flow anchored in the design system, and tools sorted by maturity level.

THE PROBLEM

Once the design system was in place, production remained. Traditional screen-by-screen mockup creation is slow when a product has multiple audiences (athlete, coach, owner) and several themes to adapt. Conversely, asking an AI to generate an interface "in a vacuum" produces generic results: plausible screens but off-system, ignoring components and tokens. Neither approach offered both speed and consistency.

WHAT I DID

I built a workflow where AI never starts from scratch: it generates based on the design system and the product's codebase.

  • The DS as an anchor point: organized by semantic tokens per theme, ensuring the generated screens are truly grounded in the system, not approximations to be reworked.
  • A refined process with Claude Code, faithful to real components and tokens. The AI proposes, I refine as a designer, the mockup moves to conception; implementation remains with the developers.
  • Automated Figma file maintenance via Figma MCP driven by scripts generated with Claude: a repetitive task becomes fast and consistent.
  • Control at every step: I define the intent and constraints, AI accelerates, I keep the decision-making and quality control.
WHAT I'M EXPLORING NEXT · POC

I'm testing interface generation with Claude Design, integrating the design system mirrored from the repo to rely on real components. The goal remains constant (never generate off-system), but the tooling evolves. I present this as an ongoing exploration, not a given: distinguishing the established from what I'm testing is part of my way of working.

THE DECISION I DARE MAKE

Anchoring generation on the design system rather than prompting in a vacuum

The common reflex is to describe a screen to the AI and hope for a good result. The issue: it's disconnected from the system, everything has to be reworked, and the speed gain vanishes. I made the opposite choice: rely on the already built DS so the AI generates within it, not beside it. This is only possible because the foundation existed and was deeply structured.

The second stance is the designer's role. AI isn't a substitute, it's a fast executor: the judgment (what's right for the user, what we keep) remains with the designer.

A TOOL TESTED THEN DISCARDED

Bridge, evaluated then abandoned

I evaluated Bridge to generate directly in Figma. After putting it to the test, I set it aside: production and iteration were too slow, token consumption too high compared to the gain. Testing a tool and deciding not to keep it is part of the job.

THE OUTCOME

The workflow accelerates production while preserving design intent and system consistency. A real process, not a demo: anchored in a production DS and used daily. The boundary remains clear: I design and generate mockups anchored in the system, implementation is up to the developers.

02 HUSTLE UP

Hustle Up design system

Systematizing a disparate existing base to support three audiences and multiple themes, without duplicating it.

TL;DR

Semantic token architecture across 4 themes (Athlete Light/Dark, Owner, Coach). A component never references a color, only a role. The same button changes appearance depending on context without duplication. Integrated with devs via Storybook.

Design, documentation, and dev integration
Figma (and Figma MCP), Storybook
late 2024
ONE COMPONENT · 4 THEMES

Anatomy of a component, applied across four themes.

THE PROBLEM

Hustle Up targets three audiences (athletes, owners, coaches), each with its own theme and accent, on web and mobile, light and dark. The starting point wasn't a blank page: components existed, but were disparate, with no tokens and incomplete color management. The real challenge: making a single base support all these contexts without duplicating it, starting from an uneven existing foundation.

WHAT I DID
  • A token architecture where there was none: global colors at the base, then a semantic layer (interactive, surface, feedback) describing intent, not value.
  • Reworked existing components to remove disparities and complete color management, ensuring everything aligns under the same tokens.
  • Adapted across four themes (Athlete Light, Athlete Dark, Owner, Coach): the same token points to a different value depending on the theme, without touching the component.
  • Documented in Storybook and integrated with devs via a shared design ↔ code token system: no manual translation, no gap between mockup and product.
LIVE · TOKEN INSPECTOR
INSPECT · CONSUMED TOKENS
  • interactive/background/primary
  • interactive/text/on-primary
  • surface/background/default

The button only references roles. Switch themes: its values follow, its code doesn't change. Hover a token to see what it paints.

THE DECISION I DARE MAKE

A semantic token layer rather than direct colors

The simple choice would have been assigning colors directly to components, theme by theme. It works at first, but becomes unmanageable as soon as you add an audience or a dark mode. I inserted a semantic layer between global colors and components: a button doesn't say "I am purple," it says "I use the primary interactive color," and the theme decides.

This level of indirection requires more upfront work, but it's exactly what makes the multi-audience system sustainable, and what made plugging in an AI-assisted workflow possible.

SCOPE COVERED

Individual coach tracking

Goals, cycles, injuries, events, and records with their progression.

Timer & BPM tracking

Heart rate tracking in the timer app, curve and averages.

Social space

Exchanges around a session, cards adapted across all their states.

CrossFit box section

Athlete view: info, slot booking, coach contact, shop.

Coach showcase

Highlighting coaches to sell training and tracking.

THE OUTCOME

Hustle Up has a design system that supports its three audiences and themes on a single base, where adding a universe doesn't break components. Most importantly, it lives within the product thanks to shared design ↔ code tokens and the integration led with the devs on Storybook. And this is the same foundation that later allowed grafting the AI-assisted workflow.

03 HUSTLE UP

Programming header redesign

Putting the reading of a dense page back in the right order.

TL;DR

The programming header had the title centered, a contrasted area that short-circuited the gaze, and two scattered visualization controls. Reworked based on the eye's path, becoming a pattern reused on coach views.

UI/UX designer
Session programming view
Reused on coach planning views
EYE TRACKING

The old header: drawn by the contrasted area in the center, the gaze zigzags instead of following the reading flow.

THE PROBLEM

The programming view header accumulated too many elements, and its structure hindered reading. Three problems compounded:

  • The eye scans the page from left to right, yet the program title was relegated to the center, lost in the middle rather than at the top.
  • A high-contrast area right in the middle short-circuited natural reading.
  • Two places were used to change the visualization: the date on the left, density on the right via a switch—an inconsistent component for toggling between two modes.
WHAT I DID

Rather than tweaking the existing layout, I reworked the header starting from the eye's path: the title returns to the top, the high-contrast area is softened, and the visualization control is made coherent instead of being split between a date and a switch.

THE DECISION I DARE MAKE

Making the header follow the page's reading flow

On a dense page, a header isn't a toolbar to fill; it's what provides the entry point for the gaze. The program title returns to where the eye first lands, the contrasted area stops interrupting the reading flow, and choosing how we view the week becomes a single logic instead of two. It must follow the reading flow, not oppose it.

THE OUTCOME

The header now guides reading instead of hindering it. The gained consistency made it a reusable base: it was adopted as-is for the new coach planning views, avoiding reinventing a header for each view.

04 HUSTLE UP

Gym owner dashboard

Centralizing the daily management of a gym, with role-adapted access.

TL;DR

Owner interviews → single management view answering "where do I act today?". Each block = an entry point to details, not the detail itself. Finance segmented by role. Need confirmed in a debrief interview (perceived importance: 5/5); an unused widget removed post-launch.

DS, user research, view design
1 PO, and the CEO for some feedback
Figma (and Figma MCP)
late 2024

The owner's management view, dense but readable: each marker is a decision, not a feature.

THE PROBLEM

After interviewing owners, an observation: to run their gym daily, they need centralized information they either didn't have, or only found scattered across the product: attendance, class fill rates, engagement, and financials. The problem wasn't a lack of data, but its dispersion. Nowhere answered the morning question: is my gym running well today, and where should I take action?

WHAT I DID
  • Identified in interviews the information that matters for daily tracking and grouped the needs on a single view: today's drop-ins, trial sessions, filling classes, engagement, and a KPI section.
  • Centralized in thematic columns on the owner theme. Each block is a summary, a trend, a key figure, not an exhaustive table.
  • Accounted for roles from the design phase: finance (on the left) falls under the financial role; athlete info remains accessible to the coach, without viewing rights for finance.
  • Reused the DS unified table model for detail pages (finances, schedule, members): filters, search, export. The dashboard links to them when the owner wants to dig deeper.
THE DECISION I DARE MAKE

A management view, not an exhaustive dashboard

The temptation with a dashboard is to include everything (every athlete, every transaction) because it all already exists. I decided the opposite: the dashboard only shows what helps to decide. Each block is an entry point to a detail, not the detail itself: you see churn and the few athletes to follow up with, a link leads to the full list.

On a management interface, value isn't in the quantity displayed but in the decision it triggers. The detail must remain a click away, not right before the eyes.

IN ADDITION

Role-adapted access

One gym, multiple viewing rights. Rather than multiplying screens, the same view modulates according to the role: the financial profile sees the same structure minus the financial blocks, rearranged to remain readable. Everyone sees what concerns them, sensitive information remains framed by permissions.

THE OUTCOME

The owner has a single view that centralizes previously dispersed tracking and answers "where do I act today?". In a debrief interview on the prototype, a CrossFit box owner confirmed a pre-existing need and rated its importance 5/5; his spontaneous description of the ideal dashboard (trial sessions to follow up, prospecting, finance, planning) matches the view block for block, and he rated the perceived quality above his day-to-day tool. The same interview also flagged what didn't serve the decision: a welcome block deemed too large ("the date, I already have it on my computer"), exactly the kind of direct feedback iterations feed on.

A tracking decision highlights the method: a note-taking feature, appreciated in prototype but barely used in reality, was not maintained: keeping it would have bloated the view without aiding decisions.
